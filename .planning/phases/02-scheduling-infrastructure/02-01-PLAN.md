**deliver.yml** — Main pipeline workflow:

```yaml
name: Deliver

on:
  schedule:
    - cron: "30 1 * * *"   # 7 AM IST = 01:30 UTC
    - cron: "30 10 * * *"  # 4 PM IST = 10:30 UTC
  workflow_dispatch: {}     # Manual trigger, no inputs for Phase 2

permissions:
  contents: write           # Required for EndBug/add-and-commit to push

concurrency:
  group: deliver
  cancel-in-progress: false # Never cancel a running delivery

jobs:
  run:
    runs-on: ubuntu-latest
    timeout-minutes: 15

    steps:
      - uses: actions/checkout@v4
        # persist-credentials: true (default) — required for EndBug push

      - uses: astral-sh/setup-uv@v7
        with:
          enable-cache: true

      - name: Install dependencies
        run: uv sync --locked

      - name: Run pipeline
        run: uv run python -m pipeline.main
        # env:
        #   TELEGRAM_BOT_TOKEN: ${{ secrets.TELEGRAM_BOT_TOKEN }}
        #   GNEWS_API_KEY: ${{ secrets.GNEWS_API_KEY }}
        #   ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        #   GOOGLE_API_KEY: ${{ secrets.GOOGLE_API_KEY }}
        # Secrets commented out — activate in phases that use them

      - name: Commit state back to repo
        id: commit
        uses: EndBug/add-and-commit@v9
        with:
          add: '["data/seen.json", "data/history.json"]'
          message: "chore(state): update seen/history after pipeline run [skip ci]"
          default_author: github_actions

      - name: Log commit status
        run: echo "committed=${{ steps.commit.outputs.committed }}"
```

Key points (from research — do NOT deviate):
- Cron `30 1 * * *` and `30 10 * * *` — exactly matching success criteria, avoiding :00 peak-load delay.
- `[skip ci]` in commit message prevents workflow re-triggering loop.
- `cancel-in-progress: false` — never cancel a running delivery to avoid inconsistent state.
- EndBug `add` uses explicit paths `["data/seen.json", "data/history.json"]` — never `'.'` to avoid committing secrets.
- Secrets are commented out — they don't exist yet and would cause no harm, but the comments document what's needed for future phases.
- `timeout-minutes: 15` — fail loudly if pipeline hangs.

**keepalive.yml** — Prevents 60-day GitHub inactivity cron disable:

```yaml
name: Keepalive

on:
  schedule:
    - cron: "0 0 * * 0"    # Weekly on Sunday at 00:00 UTC
  workflow_dispatch: {}

permissions:
  actions: write             # Required for API mode

jobs:
  keepalive:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: gautamkrishnar/keepalive-workflow@v2
        with:
          use_api: true      # API touch, no dummy commit — clean history
          time_elapsed: 45   # Trigger only if 45+ days since last activity
```

Key points (from research):
- `use_api: true` (v2 default) uses GitHub API — no git history pollution.
- `time_elapsed: 45` — only fires if truly approaching the 60-day window.
- Only needs `actions: write` permission, not `contents: write`.
- Do NOT use `liskin/gh-workflow-keepalive` — disabled by GitHub Staff.
